# kasb-exaone-qlora-finetune

한국 회계기준 manager의 로컬 모델(LG EXAONE-3.5-7.8B-Instruct)을
QLoRA로 파인튜닝해서 답변 품질(근거 충실도)을 개선하려 한 3차례 실험 기록 입니다. 이후 파인튜닝 없이
프롬프트만 바꾼 대조 실험도 추가로 기록하였습니다.

## 결론부터

**파인튜닝은 3번 다 실패했다.** 파인튜닝 안 한 베이스 모델이 세 번의 재학습 시도보다 계속 더
나았다. kasb-crawler 프로덕션은 로컬 모델을 베이스(`exaone3.5:7.8b`)로 고정했고, 이 저장소는
"왜 실패했는지", "무엇을 시도했는지"를 기록해 같은 시행착오를 반복하지 않기 위한 것이다.

추가로, 학습 없이 시스템 프롬프트에 지시 1줄만 더한 대조 실험(아래 표 참조)도 해봤다 —
"답변한 문항만"의 평균으로는 Faithfulness·Answer Relevancy가 베이스라인보다 낮지만, 거부(=사용자
입장에서 가치 0)까지 반영해 **30문항 전체 기준**으로 다시 계산하면 베이스라인·파인튜닝 3회
전부를 앞선다(자세한 계산은 아래 "참고 실험" 참조). **학습이라는 비싼 수단 없이도 지금까지의
모든 시도(베이스라인 포함)보다 전체적으로 더 유용한 결과가 가능하다**는 근거. 같은 프롬프트를
파인튜닝판 v3에 추가로 얹어봐도(거부율은 전 실험 통틀어 최저인 10.0%까지 더 떨어졌지만)
프롬프트 단독판을 이기지는 못했다 — **이 문제엔 파인튜닝보다 프롬프트 설계가 더 비용 대비
효과적인 개입**이라는 근거가 하나 더 늘었다.

## 평가 방법 (전 실험 공통)

- **held-out 30문항**: `eval/goldenset.jsonl`(전체 1195문항, 학습 데이터에서 제외된 순수 평가용)에서
  게시판별 비례 샘플링(seed=42)한 30문항. 베이스라인/v1/v2/v3와 아래 프롬프트 실험까지 전부 동일 샘플.
- **판사**: GPT-4o-mini, 전 실험 동일 모델(비교 일관성 유지). `rag/eval/judge.py`의
  Faithfulness(답변 주장이 근거자료로 뒷받침되는 비율, 0~1) / Answer Relevancy(질문에 실제로
  답했는지, 0~1) 두 지표.
- **거부율**: 30문항 중 모델이 "근거를 찾지 못했습니다"로 답변을 포기한 비율.
- **Faithfulness/Answer Relevancy는 실제로 답변한(거부하지 않은) 문항만의 평균** — 거부한 문항은
  애초에 채점 대상(답변) 자체가 없음. 분모(거부율)와 분자(Faithfulness) 기준이 다르다는 점 주의.
- 베이스라인 0.900/0.975는 재현성까지 확인함(같은 grounding으로 판사만 재호출 → 20/20 문항 점수
  완전 동일, `eval/results/exaone_baseline_rejudge.jsonl` 참조).

## 결과 비교

아래 Faithfulness·Answer Relevancy는 **30문항 전체 기준(거부한 문항은 0점으로 반영)** 값이다 —
"답변한 문항만"의 조건부 평균은 거부를 분모에서 빼버려 그 손실을 가려버리므로, 실제로 사용자에게
전달되는 가치를 보려면 거부율까지 곱해야 한다(계산 방식은 아래 "참고 실험" 및
[`docs/RESULTS.md`](docs/RESULTS.md) 참조). 버전별 "답변한 문항만" 조건부 수치도 그쪽에 그대로 있다.

| | 거부율 | Faithfulness(전체 30문항, 거부=0점) | Answer Relevancy(전체 30문항, 거부=0점) |
|---|---|---|---|
| 베이스라인 (파인튜닝 안 함) | 33.0% (10/30) | 0.600 | 0.650 |
| v1 (표준 SFT) | 66.7% (20/30) | 0.267 (최저) | — (원 지표 자체 미측정) |
| v2 (completion-only loss + eval_loss 체크포인트 선택) | 46.7% (14/30) | 0.400 | 0.343 (최저) |
| v3 (데이터 8개 통일 + 생성·판사 기반 체크포인트 선택) | 43.3% (13/30) | 0.450 | 0.410 |
| 베이스 + 프롬프트 1줄 추가(학습 없음, 아래 참조) | 13.3% (4/30) | 0.744 | **0.807** (지금까지 최고) |
| **v3 + 프롬프트 1줄 추가**(파인튜닝+프롬프트 결합, 아래 참조) | **10.0%** (3/30, 지금까지 최저) | **0.750** (지금까지 최고) | 0.647 |

이 기준으로도 v3는 v1·v2보다 낫지만 베이스라인은 못 넘고, 베이스·v3 둘 다 "프롬프트 1줄 추가"를
얹은 쪽이 원본보다 전부 앞선다. 버전별 상세 원인·하이퍼파라미터·데이터 구성은
[`docs/RESULTS.md`](docs/RESULTS.md) 참조.

## 참고 실험 — 파인튜닝 없이 프롬프트 1줄만 추가 (2026-07-22)

파인튜닝 3회가 전부 베이스라인을 못 넘은 뒤, "파인튜닝 자체를 접고 프롬프트만 바꾸면 어떨까"를
가장 저비용으로 먼저 확인해본 대조 실험. `rag/graph.py`는 전혀 수정하지 않고, 평가 스크립트
안에서 `Pipeline._answer_system_prompt`를 런타임에만 monkey-patch해 실제 서비스 프롬프트 끝에
지시 1개만 추가했다(긍정 지시만 — 부정 지시가 refusal을 급증시킨다는 기존 교훈 반영):

> "핵심 결론을 말하기 전에, 그 결론의 근거가 되는 근거 원문 문장을 [식별자] 인용과 함께 그대로
> 한 번 옮겨 적은 뒤 결론을 서술하라(원문을 바꿔 쓰지 말 것)."

출처는 RAFT 논문(Zhang et al. 2024)의 CoT+원문축어인용 포맷 — `scripts/raft_data_gen_v3_fullregen_unused.py`의
`good_sys`(학습데이터 생성용 프롬프트)에 이미 구현돼 있었지만 한 번도 실전에 쓰인 적 없던 문장을,
"학습데이터 생성용"이 아니라 "실전 답변용"으로 그대로 옮겨 **학습 없이** 베이스 모델에 적용했다.

**결과 — "답변한 문항만" 기준으로는 하락, "전체 30문항 기준"으로는 베이스라인도 능가**:

| | 거부율 | Faithfulness(답변한 문항만) | Answer Relevancy(답변한 문항만) |
|---|---|---|---|
| 베이스라인 → 프롬프트 추가 | 33.0% → **13.3%**(개선) | 0.900 → 0.858(하락) | 0.975 → 0.931(하락) |

거부율은 베이스라인보다도 낮아졌지만(구체적으로 "뭘 할지"를 시켜서 모델이 덜 포기함), 답변한
문항 26건 중 8건(30.8%)에 근거 없는 내용(unsupported claim)이 섞여 "답변한 문항만"의 평균
Faithfulness·Answer Relevancy는 둘 다 베이스라인보다 낮아졌다.

다만 이 조건부 평균은 거부한 문항을 아예 분모에서 빼버려서, "거부=가치 0"이라는 사실을 반영하지
못한다. 거부율까지 곱해 **30문항 전체 기준**으로 다시 계산하면 그림이 달라진다:

| | Faithfulness(전체 30문항, 거부=0점) | Answer Relevancy(전체 30문항, 거부=0점) |
|---|---|---|
| 베이스라인 | 0.900×66.7% = **0.600** | 0.975×66.7% = **0.650** |
| 프롬프트 추가 | 0.858×86.7% = **0.744** | 0.931×86.7% = **0.807** |

**이 기준으로는 프롬프트 추가판이 베이스라인까지 포함해 전부를 이긴다.** "답변 품질은 살짝
떨어지지만 훨씬 덜 포기한다"가 "품질은 최상이지만 1/3은 아예 도움을 못 준다"보다 사용자 입장에서
더 유용하다는 뜻 — 파인튜닝 3회(v1/v2/v3)는 물론 원본 베이스라인 대비로도, 프롬프트 1줄 추가가
전체적으로는 더 나은 선택이라는 근거다.

부수 확인: 이 실험에서 golden 근거 문단이 실제 검색(top-8)에 잡힌 비율은 8/30(26.7%)로 v2·v3와
정확히 일치 — 검색 파이프라인이 어떤 로컬모델/프롬프트를 쓰든 동일하게 동작한다는 걸 실측으로
최종 확인.

재현: `eval/exaone_baseline_quotefirst_eval.py`, 원시 결과 `eval/results/exaone_baseline_quotefirst_results.jsonl`.

### v3(파인튜닝판)에 같은 프롬프트를 얹으면? (2026-07-22)

v3 LoRA 모델에도 동일한 프롬프트 1줄을 얹어 같은 30문항으로 재평가했다(`local_model="exaone3.5-lora:v3"`만
다르고 나머지 조건 전부 동일).

| | 거부율 | Faithfulness(답변한 문항만) | Answer Relevancy(답변한 문항만) |
|---|---|---|---|
| v3(원본 프롬프트) | 43.3% (13/30) | 0.794 | 0.724 |
| v3 + 프롬프트 1줄 | **10.0%**(3/30, 전 실험 통틀어 최저) | 0.833 | 0.719 |

v3 단독보다는 거부율·Faithfulness 둘 다 개선(Relevancy는 거의 그대로). 다만 전체 30문항 기준
(거부=0점)으로 베이스+프롬프트와 비교하면 결과가 엇갈린다:

| | Faithfulness(전체 30문항) | Answer Relevancy(전체 30문항) |
|---|---|---|
| v3 + 프롬프트 1줄 | **0.750** | 0.647 |
| 베이스 + 프롬프트 1줄 | 0.744 | **0.807** |

**결론**: 파인튜닝을 프롬프트에 추가로 얹는 게 손해는 아니지만(v3 단독보다 전부 개선, 거부율은
전 실험 최저), 프롬프트만 쓴 베이스 모델을 이기지는 못한다 — 특히 "질문에 실제로 답했는가"를
재는 Answer Relevancy에서 크게 뒤진다(0.647 vs 0.807). 즉 **이 문제엔 파인튜닝보다 프롬프트
설계 쪽이 훨씬 비용 대비 효과적인 개입**이라는 근거가 하나 더 늘었다. 상세 계산과 해석은
[`docs/RESULTS.md`](docs/RESULTS.md) 참조.

재현: `eval/exaone_v3_quotefirst_eval.py`, 원시 결과 `eval/results/exaone_v3_quotefirst_results.jsonl`.

## 저장소 구성

```
notebooks/   Colab 학습 노트북 원본 3개(v1/v2/v3, 그대로 재실행 가능)
scripts/     노트북 빌드 스크립트(JSON 셀 surgery) + 학습데이터 생성/패치 스크립트
eval/        held-out 30문항 평가 스크립트 + 원시 결과(jsonl)
data/        실제 학습에 쓴 RAFT 스타일 데이터 (v1/v2용 700건, v3 패치본 700건)
docs/        버전별 상세 분석(하이퍼파라미터·데이터 구성·실패 원인)
```

## 재현 방법

1. `scripts/raft_data_gen_v1v2.py` (v1/v2) 또는 `data/raft_train_data_v3.jsonl`(이미 생성됨, v3)로
   학습 데이터 준비. GPT 호출이 필요한 건 v1/v2용 생성 스크립트뿐 — v3는 로컬 재검색 패치만이라
   비용 없음(`scripts/raft_data_patch_v3.py`).
2. `notebooks/v{1,2,3}_kasb_exaone_qlora_colab.ipynb`를 Google Colab에 업로드해 위에서부터 순서대로 실행.
3. 결과 GGUF를 Ollama에 배포(`ollama create`, 노트북 14단계 안내 참조) 후
   `eval/exaone_v{2,3}_final_eval.py` 또는 `exaone_baseline_eval.py`로 held-out 30문항 평가.
